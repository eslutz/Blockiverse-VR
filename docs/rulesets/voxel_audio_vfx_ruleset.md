# Voxel Audio and VFX Ruleset

**Document status:** Proposed implementation ruleset
**Project:** Blockiverse VR
**Primary target:** Meta Quest 3 / Quest 3S
**Scope:** Audio cues, haptics, visual effects, environmental presentation, multiplayer feedback behavior, and Quest performance rules

This document defines how Blockiverse VR should present audio, haptics, particles, transient visual effects, and environmental feedback. It treats event-driven presentation as the canonical architecture: gameplay systems raise authoritative events, while audio, haptics, particles, subtitles, and accessibility feedback subscribe without changing simulation logic.

---

## 1. Design goals

| Goal | Rule |
|---|---|
| Original identity | All audio, VFX, music, and visual motifs must be original Blockiverse assets. Do not imitate protected voxel-game sounds, music, UI, mobs, or branding. |
| VR comfort | Avoid abrupt full-screen flashes, excessive camera-space particles, loud spikes, or haptic patterns that can cause discomfort. |
| Decoupled presentation | Gameplay code raises events; audio, haptics, and VFX subscribe as presentation systems. |
| Quest performance | Effects must be pooled, short-lived, low-overdraw, and bounded by budgets suitable for Quest 3/3S. |
| Multiplayer correctness | Play authoritative gameplay feedback only after local authoritative events or accepted host deltas. |
| Accessibility | Provide volume sliders, mute options, haptic intensity controls, reduced-flash mode, and non-audio UI feedback for important events. |

---

## 2. Event-driven presentation baseline

Baseline feedback systems:

| System | Required Behavior |
|---|---|
| Audio cue enum | `BlockBreak`, `BlockPlace`, `UiSelect`, `UiConfirm`, `UiCancel`, `Footstep`, `InventoryOpen`, `InventoryClose`, `CraftSuccess`, `CraftFail`, tool/contact, container, fire, weather, ambience, and multiplayer join/leave cues. |
| Cue player | One central `BlockiverseAudioCuePlayer` plays one-shot audio clips, supports pooled world-space one-shots, and subscribes to block mutation events. |
| Generated clips | Original synthesized WAV cues are generated by `scripts/audio/generate-audio.py` into `Assets/Blockiverse/Audio`. |
| Haptics | Dominant-hand haptics subscribe to block mutation events, expose UI tick feedback, and scale through player feedback settings. |
| VFX | `BlockiverseVfxCuePlayer` subscribes to the same feedback events and uses `BlockiverseVfxPool` for short-lived pooled particle systems. |
| Voice | No in-app voice. Use Meta Quest party chat for cooperative multiplayer communication. |

## 3. Feedback event model

Presentation systems should use a shared event bus or a set of small typed events. Gameplay systems must not directly instantiate particles, audio sources, or haptic impulses except through presentation-facing interfaces.

### Core feedback event schema

```ts
type FeedbackEvent = {
  eventId: string;
  category: "block" | "ui" | "inventory" | "crafting" | "movement" | "weather" | "survival" | "multiplayer";
  worldPosition?: Vector3;
  sourceClientId?: ulong;
  isLocalPlayerInitiated: boolean;
  isAuthoritative: boolean;
  blockChange?: BlockChange;
  itemId?: string;
  intensity?: number;      // 0.0 to 1.0
  surfaceTag?: string;
  context?: Record<string, string | number | boolean>;
};
```

### Event source rules

| Source | Event Rule |
|---|---|
| Block edits | Raise feedback after a mutation is locally applied from host authority or single-player host path. |
| Client pending request | May show preview-only UI/haptic feedback, but must not play final placement/break confirmation until host delta arrives. |
| UI buttons | Play UI cues immediately on local selection/confirm/cancel. |
| Crafting | Play success/fail after local host result or received command result. |
| Inventory | Play open/close locally when menus open/close. |
| Footsteps | Play locally from locomotion cadence; remote footsteps use remote pose speed with distance attenuation. |
| Weather | Weather manager emits ambient audio/VFX state changes from authoritative or saved world environment state. |
| Multiplayer disconnect | UI status event plays non-spatial cancel/warning cue and optional haptic tick. |

### Presentation subscribers

```txt
BlockiverseAudioCuePlayer
BlockiverseInteractionHaptics
BlockiverseVfxCuePlayer        // new
BlockiverseSubtitleToastPanel  // optional accessibility layer
PerformanceStatsOverlay        // development only; not player feedback
```

## 7. Audio asset rules

### Originality and provenance

| Rule | Requirement |
|---|---|
| No protected references | Do not use or imitate Minecraft sounds, music, UI cues, mob cues, or branding. |
| Generated baseline | Generated cues must remain reproducible from the committed audio generator script. |
| Authored replacement | Hand-authored replacement sounds are allowed only with clear original provenance. |
| Third-party assets | Use only if redistribution and commercial use are explicitly allowed; record license in notices. |
| File naming | Lowercase snake_case, e.g. `block_break.wav`. |
| Metadata | Unity `.meta` files are committed. |
| Large files | Track `.wav` files through Git LFS. |

### Recommended import settings

| Setting | Value |
|---|---|
| Format | WAV for source; platform compression may be added after headset testing. |
| Channels | Mono for one-shot SFX. |
| Sample rate | 44.1 kHz source unless a specific reason exists. |
| Peak target | Approximately -1.7 dBFS / 0.82 peak for generated cues. |
| Normalize | Enabled for generated baseline cues. |
| Preload | Enabled for short SFX. |
| Load in background | Disabled for short SFX. |
| Spatial | Import itself non-spatial; spatial behavior is controlled by the AudioSource. |

## 8. Playback rules

### 2D vs spatial playback

| Cue Type | Playback Rule |
|---|---|
| UI, inventory, crafting menu | 2D/non-spatial. |
| Local block edit | Prefer short spatial one-shot at block center; 2D fallback allowed only until pooled spatial sources exist. |
| Remote block edit | Spatial one-shot at edited block position, distance-limited. |
| Footsteps | Local comfort-tuned low-volume cue; remote footsteps spatial and distance-limited. |
| Weather | Area/ambient loops mixed from weather state. |
| Thunder | 2D + spatial hybrid; distance delay and volume based on strike distance. |
| Campfire/torch | Spatial loop or periodic one-shot with strict voice limits. |

### Voice limits

| Category | Max Simultaneous Voices |
|---|---:|
| UI | 4 |
| Block edits | 8 |
| Footsteps | 4 |
| Weather loops | 3 |
| Ambience loops | 2 |
| Fire/light loops | 6 nearest sources |
| Total target | 24 active voices or fewer |

### AudioSource pooling

Do not create/destroy AudioSource objects per event.

```ts
class AudioOneShotPool {
  AudioSource[] sources;
  int nextIndex;

  void Play(AudioClip clip, Vector3? position, AudioMixerGroup bus, float volume) {
    source = FindAvailableOrStealLowestPriority();
    source.clip = null;
    source.outputAudioMixerGroup = bus;
    source.spatialBlend = position.HasValue ? 1.0 : 0.0;
    if position.HasValue: source.transform.position = position.Value;
    source.PlayOneShot(clip, volume);
  }
}
```

### Distance attenuation

| Cue | Max Distance | Rolloff |
|---|---:|---|
| Block break/place | 12 m | Linear or logarithmic, gentle. |
| Remote footsteps | 8 m | Linear. |
| Container open/close | 8 m | Linear. |
| Campfire loop | 10 m | Logarithmic. |
| Torch loop | 4 m | Very subtle. |
| Thunder near | Global with distance-based delay. | Custom. |

## 9. Haptics rules

### Current patterns

| Pattern | Amplitude | Duration | Trigger |
|---|---:|---:|---|
| `BlockBreak` | 0.60 | 0.05s | Authoritative block removed by local player. |
| `BlockPlace` | 0.40 | 0.04s | Authoritative block placed by local player. |
| `UiTick` | 0.25 | 0.02s | UI confirm/cancel/select. |
| `CraftSuccess` | 0.28 | 0.04s | Crafting accepted. |
| `CraftFail` | 0.18 | 0.06s | Crafting rejected. |

### Haptic scaling

```ts
finalAmplitude = baseAmplitude * userHapticIntensity;
if hapticsDisabled:
  finalAmplitude = 0;
```

### Haptic ownership rules

| Event | Controller |
|---|---|
| Block edit | Dominant hand. |
| UI interaction | Pointer hand if known; dominant hand fallback. |
| Inventory open/close | Dominant hand light tick. |
| Craft success | Both hands optional, low amplitude. |
| Craft fail | Dominant hand low warning pulse. |
| Weather/thunder | Off by default; optional accessibility-safe low rumble only. |

Avoid long haptic patterns. Any pattern longer than `0.12s` requires explicit comfort review.

## 10. VFX architecture

VFX is implemented through a presentation component rather than changing block-edit logic.

### Component

```ts
public sealed class BlockiverseVfxCuePlayer : MonoBehaviour {
  [SerializeField] CreativeInteractionController interactionController;
  [SerializeField] BlockiverseVfxPool pool;

  void OnEnable() {
    interactionController.BlockMutationApplied += OnBlockMutationApplied;
  }

  void OnBlockMutationApplied(BlockChange change) {
    if change.NewBlock == BlockRegistry.Air:
      Play(BlockiverseVfxCue.BlockBreakDust, Center(change.Position));
    else:
      Play(BlockiverseVfxCue.BlockPlacePuff, Center(change.Position));
  }
}
```

### VFX cue enum

```ts
enum BlockiverseVfxCue {
  BlockBreakDust,
  BlockPlacePuff,
  BlockChipBurst,
  ResourceSpark,
  CraftSuccessSpark,
  CraftFailPuff,
  InventoryPanelPulse,
  SelectionPulse,
  TeleportArrivalPuff,
  RainSplash,
  SnowflakeDrift,
  FogWisp,
  LightningFlash,
  TorchSpark,
  CampfireEmber
}
```

### VFX pooling rules

| Rule | Value |
|---|---:|
| No instantiate/destroy during gameplay events | Required. |
| Default pooled one-shot systems | 32 small systems. |
| Weather particle pools | Separate from one-shot pool. |
| Max active one-shot particle systems | 16. |
| Max particles per block break | 12. |
| Max particles per block place | 8. |
| Max lifetime for block edit particles | 0.45s. |
| Max transparent screen coverage | Keep low; avoid full-screen particle sheets. |

## 12. Environment audio/VFX rules

Environment presentation should be driven by the world environment state rather than hard-coded scene effects.

### Environment state input

```ts
type EnvironmentPresentationState = {
  timeOfDay01: number;
  lightLevel: number;
  cloudCoverage01: number;
  precipitation: "none" | "rain" | "snow";
  precipitationIntensity01: number;
  thunderstormIntensity01: number;
  fogDensity01: number;
  windIntensity01: number;
  biomeId: string;
  isPlayerUnderground: boolean;
  isPlayerSheltered: boolean;
};
```

### Day/night ambience

| Condition | Audio | VFX/Lighting |
|---|---|---|
| Day | Light biome ambience. | Warmer sky and brighter directional light. |
| Sunset/sunrise | Gentle transition layer. | Warm horizon tint. |
| Night | Lower ambience, subtle nocturnal layer. | Cooler sky and lower global light. |
| Underground | Cave ambience overrides most surface ambience. | Reduced sky/weather VFX. |

### Cloud coverage

| Cloud Coverage | Presentation |
|---:|---|
| 0.0–0.25 | Clear sky, full sunlight. |
| 0.25–0.65 | Soft light dimming, sparse cloud forms. |
| 0.65–0.90 | Overcast, muted sunlight, possible rain layer. |
| 0.90–1.00 | Storm lighting, darker sky, thunder possible. |

### Rain

| Rule | Value |
|---|---:|
| Outdoor rain loop fade-in | 2–5 seconds. |
| Indoor/sheltered rain volume | 20–40% of outdoor volume. |
| Max visible raindrops | 150–250 camera-relative particles on Quest. |
| Splash particles | Only near gaze/player area, max 30 active. |
| Lightning allowed | Only when thunderstorm intensity > 0.6. |

### Thunder and lightning

```ts
if thunderstormIntensity > 0.6 and randomStormTimerElapsed:
  strikePosition = chooseOutdoorPointNearPlayerOrDistantHorizon()
  flashDelay = 0
  thunderDelay = distance(player, strikePosition) / 343.0
  PlayLightningFlash(strikePosition)
  ScheduleThunder(thunderDelay)
```

Comfort rules:

| Setting | Effect |
|---|---|
| Reduced Flash disabled | Lightning flash may briefly raise sky/key light. |
| Reduced Flash enabled | Use muted sky pulse only; no high-contrast full-screen flash. |
| Weather volume low | Thunder respects weather bus volume. |
| Haptics disabled | No thunder rumble. |

### Fog

| Fog Density | Presentation |
|---:|---|
| 0.0–0.2 | No visible fog particles; optional URP fog only. |
| 0.2–0.5 | Light distance fog; no near-camera wisps. |
| 0.5–0.8 | Thicker distance fog and sparse low wisps. |
| 0.8–1.0 | Dense fog; cap visibility effects to preserve comfort and navigation. |

### Snow

| Rule | Value |
|---|---:|
| Snow particle density | 80–160 visible flakes. |
| Flake speed | Slow; avoid rapid camera-relative streaks. |
| Accumulation VFX | Prefer material/terrain state changes over many particles. |
| Snow audio | Soft wind layer; no constant high-frequency hiss. |
| Shelter handling | Reduce or hide flakes when player is under solid cover. |

## 14. Multiplayer feedback rules

| Event | Local Initiator | Other Clients |
|---|---|---|
| Client places block | Preview may show immediately; final audio/VFX/haptic after host delta. | Audio/VFX after host delta, spatial at block. |
| Host places block | Final audio/VFX immediately after commit. | Audio/VFX after received delta. |
| Client rejected edit | UI cancel/fail cue only for requesting client. | No feedback. |
| Client harvest accepted | Success/drop cue for requester; block break feedback from delta. | Spatial block break feedback from delta. |
| Craft accepted | Success cue for requesting player only. | No global cue unless station is shared and visible. |
| Shared crate transfer | UI cue for requester; crate UI refresh for viewers. | Optional subtle UI refresh only. |
| Player join/leave | UI/system cue for all connected players. | Same. |
| Remote footsteps | Driven by remote pose speed and distance. | Local player does not hear duplicate local footstep. |

### Prediction rule

Do not play irreversible confirmation feedback before host acceptance. Acceptable pre-confirmation feedback:

```txt
placement preview outline
light controller hover haptic
soft UI select tick
pending spinner/toast
```

Not acceptable before host acceptance:

```txt
final block place thump
final block break crumble
resource pickup sound
craft success cue
inventory item count animation
```

## 15. Accessibility rules

| Setting | Required Behavior |
|---|---|
| Master Volume | Scales all in-game audio. |
| Effects Volume | Scales gameplay one-shots. |
| UI Volume | Scales menu feedback. |
| Weather Volume | Scales rain/thunder/wind. |
| Mute All | Mutes all game audio except platform/system audio outside game control. |
| Haptics Toggle | Disables all game-triggered haptics. |
| Haptic Intensity | Scales amplitudes 0–100%. |
| Reduced Flash | Reduces lightning and intense VFX pulses. |
| Reduced Particles | Lowers particle counts for weather and block effects. |
| Feedback Toasts | Shows text/icon confirmation for important audio-only events. |

Important events should have at least two feedback channels when practical:

```txt
craft fail: sound + UI text
host disconnected: sound + status text
inventory full: sound + UI text
block edit disabled: haptic/UI hint + status icon
```

## 16. Quest performance budgets

| Budget | Target |
|---|---:|
| Minimum frame rate | 72 FPS stable. |
| Optimization target | 90 FPS where feasible. |
| Runtime allocations per feedback event | 0 B after warmup. |
| One-shot particle systems active | ≤ 16. |
| Weather particles visible | ≤ 250 rain or ≤ 160 snow for the initial Quest budget. |
| Total active audio voices | ≤ 24. |
| Long-running spatial loops | ≤ 8 nearest relevant loops. |
| Lightning flash duration | ≤ 0.12s, reduced by accessibility setting. |
| Full-screen overdraw effects | Avoid for Quest comfort and performance. |
| Particle collision | Avoid except cheap rain splash approximations. |

### Pooling requirements

```txt
Preload common clips before gameplay.
Prewarm particle pools on scene load.
Never allocate new materials per effect event.
Use shared materials with per-instance property blocks where possible.
Cull effects outside audible/visible range.
Stop weather effects when player is sheltered or underground unless needed audibly.
```

## 19. Acceptance criteria

A feature using this ruleset is complete when:
