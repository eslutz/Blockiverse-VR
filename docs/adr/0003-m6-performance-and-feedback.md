# ADR 0003: Performance Instrumentation and Interaction Feedback

## Status

Accepted

Current design authority for audio, haptics, VFX, multiplayer feedback, and Quest effect budgets now lives in [Voxel Audio and VFX Ruleset](../rulesets/voxel_audio_vfx_ruleset.md). This ADR records the repository decision for performance instrumentation and interaction feedback architecture.

## Decision

Implement performance instrumentation and interaction feedback as follows:

- **Performance instrumentation.** Wrap the hot CPU paths (canonical world generation,
  chunk meshing, and renderer rebuilds) with named `Unity.Profiling.ProfilerMarker`s. Add an
  engine-free `FrameStatisticsSampler` in Core for unit-testable frame timing, surfaced through
  a local-only `PerformanceStatsOverlay` that reuses the existing `VoxelRenderStats` and the
  `BlockiverseLog` `Performance` category. The overlay never renders in release builds.
- **CPU performance proxy.** Cover the largest shipped canonical world preset
  with an EditMode stress test that generates and meshes every chunk and asserts the work is
  deterministic and bounded. On-headset frame-rate evidence is recorded separately under
  `docs/testing/performance` using a report template.
- **Interaction feedback.** Raise a single `BlockMutationApplied` event from
  `CreativeInteractionController` on applied edits/undo. Audio (`BlockiverseAudioCuePlayer`,
  Gameplay) and haptics (`BlockiverseInteractionHaptics`, VR) subscribe to that event so
  presentation stays decoupled from edit logic. The same cue player also handles UI,
  inventory/crafting, and teleport footstep/landing feedback so the project has one generated,
  original sound-effect set. Sound effects are original synthesized cues generated by
  `scripts/audio/generate-m6-audio.py`.

## Context

Blockiverse needs local performance visibility and sensory feedback polish before release
candidate validation. This decision scopes the repository architecture for instrumentation,
performance evidence, audio, and haptics. Release packaging, production signing, Meta
dashboard actions, and platform submission remain separate workflows.

Profiler markers and the in-game stats panel were explicitly deferred from the Alpha logging
foundation (see ADR 0002) to this dedicated instrumentation and feedback decision.

## Consequences

- Profiler markers add negligible overhead and are safe to leave compiled in.
- The performance overlay is gated to development builds and the editor, satisfying the VRC
  requirement that no debug-only UI ships in release.
- Feedback systems depend only on the public `BlockMutationApplied` event, so additional
  responders (e.g. particles) can be added without touching edit logic.
- Audio is produced by `scripts/audio/generate-m6-audio.py` into
  `Assets/Blockiverse/Audio` and tracked via Git LFS (`*.wav`). The generated cues remain
  reproducible and original; headset acceptance evidence is tracked in
  `docs/store-submission/known-issues-and-support.md`.
