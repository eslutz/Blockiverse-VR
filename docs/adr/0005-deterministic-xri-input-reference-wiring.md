# ADR 0005: Deterministic XRI Input Reference Wiring

## Status

Accepted

## Decision

Generate the Blockiverse Unity Input System catalog deterministically from the project bootstrapper. Map, action, and binding IDs are derived from stable project-owned names instead of Unity-generated random IDs.

Track one generated `InputActionReference` asset per project-owned input action under `Assets/Blockiverse/Settings/InputActionReferences/`. Generated scenes and prefabs must reference those asset-owned references for XRI pose drivers, ray interactors, UI input modules, and default locomotion readers. They must not store scene-local `InputActionReference` objects or serialized inline `InputAction` instances for project-owned XRI actions.

Runtime repair code may preserve existing generated references, and may create transient fallback references only when it is repairing live objects that do not already have generated references. Persisted generated assets are still expected to use tracked generated references.

## Context

Unity can rewrite inline `InputAction` and scene-local `InputActionReference` serialization during import, build, or batchmode bootstrapper runs. That creates large, noisy scene and prefab diffs even when no gameplay mapping changed, and it makes review of generated XR wiring harder than it needs to be.

The project already treats Boot, the XR rig prefab, menus, and input actions as bootstrapper-generated assets. Making input IDs and reference ownership deterministic keeps that model intact and gives tests a stable contract to enforce.

## Consequences

- New input actions must be added to the bootstrapper catalog, not hand-added in the Unity editor.
- The generated `.inputactions` asset and `Assets/Blockiverse/Settings/InputActionReferences/` assets are source-controlled project assets.
- Generated scenes and prefabs should be regenerated by `Blockiverse.Editor.BlockiverseProjectBootstrapper.Run` after input changes.
- EditMode tests guard deterministic input IDs, generated reference asset coverage, generated prefab reference ownership, and the absence of scene-local input reference objects in generated scenes.
- A future migration to a different input package or hand-tracking input mode must preserve the same principle: project-owned generated assets should carry stable IDs and source-controlled references, while runtime-only fallback repair remains transient.
