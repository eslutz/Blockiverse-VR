# ADR 0005: Release Versioning

## Status

Accepted

## Context

Blockiverse VR releases to four distribution stages:

- Alpha builds from same-repository pull request commits.
- Beta builds from commits on `main`.
- Release candidates promote an existing Beta Meta build.
- Production promotes an existing RC Meta build to the Meta `store` channel.

The release process needs stable, predictable version tags that work with GitHub
Releases, Meta release channels, Android `versionName`, and human review.

## Decision

The repository root `VERSION` file is the single source of truth for the
SemVer base version. It contains only `MAJOR.MINOR.PATCH`, without a leading
`v` and without prerelease metadata.

Release tags use a leading `v` and derive from `VERSION`:

| Stage | Tag format | Example |
|---|---|---|
| Alpha | `vMAJOR.MINOR.PATCH-alpha.prPR.RUN.ATTEMPT` | `v0.1.0-alpha.pr318.42.1` |
| Beta | `vMAJOR.MINOR.PATCH-beta.runRUN.ATTEMPT` | `v0.1.0-beta.run43.1` |
| RC | `vMAJOR.MINOR.PATCH-rc.CANDIDATE` | `v0.1.0-rc.1` |
| Production | `vMAJOR.MINOR.PATCH` | `v0.1.0` |

The `VERSION` file is advanced by a normal pull request before starting a new
release train. Alpha and Beta builds are immutable GitHub Releases for the
specific commit and workflow run that produced the APK. RC and Production
releases preserve artifact identity by promoting a previously uploaded Meta
build ID instead of rebuilding.

Android `versionName` uses the GitHub release tag without the leading `v`.
Android `versionCode` is generated by CI as a monotonically increasing integer
from the workflow run timestamp. The Android code is not the product version;
it exists only to satisfy Android package upgrade ordering.

## Consequences

- Version bumps are explicit, reviewable code changes.
- Alpha and Beta prerelease tags can be created repeatedly for the same
  `VERSION` base without colliding because they include PR or workflow run
  identifiers.
- RC and Production releases cannot silently drift to a different artifact:
  they read the selected source GitHub release metadata and reuse its Meta
  build ID.
- The workflow intentionally avoids automatically inferring the next SemVer
  version from Git history or commit messages. That keeps release intent
  explicit for a small single-maintainer project.
