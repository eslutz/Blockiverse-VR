## Area Reviewed

Runtime wiring and scene/prefab composition for the Boot runtime: the committed Boot.unity, BlockiverseXRRig.prefab, BlockiverseNetworkManager.prefab, BlockiverseNetworkPlayer.prefab, MultiplayerTest.unity, EditorBuildSettings.asset, and the generating code in BlockiverseProjectBootstrapper.cs, cross-checked against the runtime classes they instantiate. Scene/prefab YAML agrees with the current bootstrapper output (the only bootstrapper change since the last regeneration — removal of a RemoveAction(Jump) line in be5ec24 — has no observable effect on the committed .inputactions). Every core system MonoBehaviour is present: CreativeWorldManager/VoxelWorldRenderer/CreativeInteractionController on the scene's Creative World; the full multiplayer stack (NetworkManager, UnityTransport, BlockiverseNetworkSession/Bootstrap, MultiplayerChunkAuthoritySync, MultiplayerSurvivalSync, SurvivalVitalsRuntime, MultiplayerWorldPersistence, EnvironmentDynamicsController) on the NetworkManager prefab; all UI panels including the new Audio Settings/Catalog Browser/Creative Tools/World Details panels, music controller, feedback stack and locomotion feedback on the XR rig prefab. Null serialized references in Boot.unity (CreativeWorldManager.placementPreview/voidSafetyFloor, BlockiverseLightingCycleController.environmentSource, XRUIInputModule point/middle/right/cancel actions) and in the NetworkManager prefab (session/worldManager/etc. on every sync component) are all covered by verified runtime fallbacks (ResolveReferences/FindFirstObjectByType or runtime creation). Input wiring is sound: scene InputActionReferences resolve to actions in BlockiverseInputActions.inputactions (GUID brace/case differences parse identically), TrackedPoseDrivers carry inline XR bindings, BlockiverseInputRig enables the asset OnEnable. No script execution order overrides exist and Awake-order races between CreativeWorldManager and the sync components are defused by mutual resolution. HOWEVER, the review found one project-breaking defect class: the bootstrapper wires the game-menu system (BlockiverseMenuController, New World/Load World panels, Station panel) by calling Configure(...) at editor time on fields that are NOT [SerializeField]-serialized, so the wiring is silently discarded when the prefab is saved — the committed prefab proves it (only stationPanel survives on the controller; the New World and Load World panel components serialize zero fields). At runtime the title/pause/death/confirm/settings routing, new-world/load-world/save flows, and station UI are dead; the title menu can never appear (its canvas is disabled, showOnStart=0, and the presenter list that would show it is empty at runtime). Overall: the simulation/feedback/network layers are wired defensively and correctly; the menu/session UI layer is structurally broken by a serialization misunderstanding in the generator.

## Findings (9)

### 1. Game menu system is wired into non-serialized fields by the bootstrapper — entire title/pause/death/session menu flow is dead at runtime

- **Severity:** Critical  |  **Confidence:** High  |  **Needs manual verification:** True
- **Files:** `Assets/Blockiverse/Scripts/UI/BlockiverseMenuController.cs`, `Assets/Blockiverse/Scripts/Editor/BlockiverseProjectBootstrapper.cs`, `Assets/Blockiverse/Prefabs/BlockiverseXRRig.prefab`
- **Impact:** On a fresh launch (built APK or any play session after a domain reload), the title menu never appears and no menu screen can function: BlockiverseMenuController's references to titleMenu/pauseMenu/deathMenu/confirmMenu/settingsMenu/newWorldPanel/loadWorldPanel/worldDetailsPanel/worldDetailsMenu/inputRig and its screenPresenters list are null/empty, so Start() wires nothing, SetMenu is never called (action menus have empty actionIds, so button clicks no-op), ApplyRouterState shows/hides no screens, and the in-game menu button never reaches OnMenuPressed. Players boot into the Awake-generated default creative world with no way to create/load/save a named world, pause, see the death screen, or respawn via UI. BlockiverseWorldSessionController subscribes to ActionRequested but no action can ever fire.
- **Evidence:** BlockiverseMenuController.cs lines 16-33: every menu/panel field is a plain private field — only `[SerializeField] BlockiverseStationPanel stationPanel;` (line 28) is serialized; `screenPresenters` is a readonly List (line 33, never serialized). The bootstrapper wires them only at editor time: BlockiverseProjectBootstrapper.cs:3874-3881 `controller.Configure(inputRig, titleMenu, pauseMenu, deathMenu, confirmMenu, newWorldPanel, loadWorldPanel, settingsMenu, worldDetailsPanel, worldDetailsMenu)` and `controller.ConfigurePresenters(...)` inside prefab generation (EnsureXrRigGameMenus, called from CreateXrRigInstance:1771 and EnsureXrRigControllerBindings:1840 before SaveAsPrefabAsset:1096/1107). Ground truth in the committed prefab: the BlockiverseMenuController MonoBehaviour block in BlockiverseXRRig.prefab serializes exactly one field, `stationPanel: {fileID: 505366546020636366}` — nothing else. Boot.unity's rig PrefabInstance (fileID 1420024869) carries no overrides for it. Runtime consequences verified in source: Start() (lines 193-216) only does `if (inputRig != null) inputRig.MenuPressed.AddListener(...)`, `WireMenus()` (527-538) subscribes only non-null menus, `ApplyRouterState()` (500-519) iterates the empty screenPresenters. Title Menu GameObject in the prefab has canvasEnabled=0 and presenter showOnStart=0 (presenter Start() only Shows when showOnStart, BlockiverseWorldSpacePanelPresenter.cs:171-175), so no screen ever becomes visible. BlockiverseActionMenu.cs:122-133 shows clicks no-op while actionIds (populated only by SetMenu) is empty. No PlayMode/EditMode test references BlockiverseMenuController, so the gap is untested.
- **Recommended fix:** Make the menu graph survive serialization: add [SerializeField] to all fields assigned by BlockiverseMenuController.Configure (titleMenu, pauseMenu, deathMenu, confirmMenu, settingsMenu, newWorldPanel, loadWorldPanel, worldDetailsPanel, worldDetailsMenu, inputRig) and replace the runtime-only screenPresenters list with a serializable structure (e.g. [Serializable] struct ScreenPresenterEntry { string screenId; BlockiverseWorldSpacePanelPresenter presenter; } + List<ScreenPresenterEntry>), or alternatively add a runtime ResolveReferences() in Awake/Start that discovers the panels by name/type under the rig (mirroring the pattern already used by MultiplayerWorldPersistence and friends). Then rerun the bootstrapper (Blockiverse → Bootstrap Unity Quest Project) so the regenerated BlockiverseXRRig.prefab actually contains the references, and add a PlayMode test that instantiates the committed rig prefab and asserts the title screen appears and a title action fires.

### 2. New World and Load World panels serialize zero control references — panels are non-functional even if the menu controller is fixed

- **Severity:** Critical  |  **Confidence:** High  |  **Needs manual verification:** True
- **Files:** `Assets/Blockiverse/Scripts/UI/BlockiverseNewWorldPanel.cs`, `Assets/Blockiverse/Scripts/UI/BlockiverseLoadWorldPanel.cs`, `Assets/Blockiverse/Scripts/Editor/BlockiverseProjectBootstrapper.cs`, `Assets/Blockiverse/Prefabs/BlockiverseXRRig.prefab`
- **Impact:** The New World screen (name/seed inputs, mode/difficulty/size/preset/biome cycle buttons, Create/Cancel) and Load World screen (save rows, Load/Details/Cancel buttons, selection label) have no references to their own UI controls at runtime: WireControls() runs over null/empty arrays, ResetForNewWorld()/SetSaves() update nothing, and ActionRequested (NewWorldCreate/LoadWorldLoad) can never fire. Single-player world creation and loading are impossible through UI regardless of the controller fix, making this a second independent break in the core session flow.
- **Evidence:** BlockiverseNewWorldPanel.cs lines 13-20: `TMP_InputField nameInput; TMP_InputField seedInput; Button[] cycleBackButtons; ... Button createButton; Button cancelButton; TMP_Text errorLabel;` — none are [SerializeField]; they are assigned only by Configure(...) which the bootstrapper calls at editor time (BlockiverseProjectBootstrapper.cs:4248 `panel.Configure(nameInput, seedInput, backButtons, nextButtons, valueLabels, ...)`). Same for BlockiverseLoadWorldPanel.cs lines 15-21 (entryButtons/entryLabels/loadButton/detailsButton/cancelButton/selectionLabel) wired at bootstrapper line 4347. Ground truth: in the committed BlockiverseXRRig.prefab, the MonoBehaviour blocks for both BlockiverseNewWorldPanel (guid 537d089e18dc6450b9b259f6be1a7f57) and BlockiverseLoadWorldPanel (guid 42f1808d583684ac3a0f3aa4511422ef) contain NO serialized fields after m_EditorClassIdentifier — empty. Contrast with sibling panels done correctly (BlockiverseAudioSettingsPanel, BlockiverseCatalogBrowserPanel, SurvivalInventoryPanel all serialize full control arrays in the same prefab).
- **Recommended fix:** Add [SerializeField] to all control-reference fields in BlockiverseNewWorldPanel and BlockiverseLoadWorldPanel (matching the pattern in BlockiverseAudioSettingsPanel/SurvivalInventoryPanel), keep Configure() for tests, move the WireControls() listener attachment into Awake() (idempotent), and rerun the bootstrapper to regenerate the rig prefab with the references persisted.

### 4. Menu button is double-bound: persistent listener toggles Comfort Settings menu while the menu controller expects to own MenuPressed for pause routing

- **Severity:** Medium  |  **Confidence:** High  |  **Needs manual verification:** True
- **Files:** `Assets/Blockiverse/Scripts/Editor/BlockiverseProjectBootstrapper.cs`, `Assets/Blockiverse/Prefabs/BlockiverseXRRig.prefab`, `Assets/Blockiverse/Scripts/UI/BlockiverseMenuController.cs`
- **Impact:** Today (with the menu controller dead) the physical menu button's only effect is toggling the Comfort Settings panel via a persistent UnityEvent call. Once the Critical menu-controller fix lands, every menu-button press will simultaneously toggle the comfort menu AND push/pop the pause screen through the router — two overlapping world-space panels and conflicting input focus on each press.
- **Evidence:** BlockiverseProjectBootstrapper.cs:2197-2208 (EnsureXrRigComfortMenu): `UnityEventTools.AddPersistentListener(inputRig.MenuPressed, presenter.ToggleVisible)` targeting the Comfort Settings Menu presenter — confirmed in the committed prefab: BlockiverseInputRig.menuPressed m_PersistentCalls targets fileID 3940017637420611929, whose GameObject is 'Comfort Settings Menu'. Meanwhile BlockiverseProjectBootstrapper.cs:3921-3924 documents 'The controller subscribes to MenuPressed at runtime (Start → AddListener)' and removes only the controller's own persistent listener, not the comfort presenter's; BlockiverseMenuController.cs:199 adds OnMenuPressed (pause routing) to the same event.
- **Recommended fix:** Pick one owner for MenuPressed. Per docs/rulesets/voxel_survival_menus.md the menu button should drive the pause/title router, so remove the comfort-menu persistent listener in EnsureXrRigComfortMenu (and surface comfort settings through the Settings screen or QuickMenu instead), then rerun the bootstrapper. At minimum, make EnsureXrRigGameMenus strip the comfort presenter's persistent listener from MenuPressed when wiring the controller.
