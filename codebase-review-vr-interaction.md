## Area Reviewed

VR/3D interaction review of Blockiverse VR on Quest 3/3S: the generated input actions asset (all three action maps and every binding), BlockiverseInputRig and its XRI provider wiring, the committed BlockiverseXRRig.prefab YAML (pose drivers, ray interactors, mediators, vignette, persistent UnityEvent listeners), the bootstrapper sections that generate the rig/menus/HUD, interaction flow (CreativeInteractionController, BlockiverseCreativeInputBridge, hold-to-mine), locomotion and comfort (glide/teleport, snap/smooth turn, tunneling vignette, height reset, settings persistence), world-space UI (panel presenter distances/scales/font sizes, menu router, survival HUD, comfort menu), camera-moving code paths, and haptic/audio/VFX feedback coverage. Overall the comfort architecture is unusually well thought out for a first playable (mode toggles, vignette plumbing, settings persistence, native system keyboard, escape-hatch menu design), but several high-impact wiring defects undermine it: the menu button drives two competing menu systems at once, the "pause" stack never actually pauses or gates world input, the motion vignette is a no-op at its default setting with an inverted slider, every interaction verb is hardcoded to the right hand with no handedness option, and the thumbstick actions ship without deadzone processors.

- **Evidence:** BlockiverseComfortSettings.cs line 24: '[SerializeField] float vignetteStrength = 1.0f' with comment 'Normalized 0–1: 1 = widest aperture (subtle), 0 = fully closed (strong)'; line 73: VignetteAperture => 0.6f + strength*0.4f → 1.0 at default. ### 4. All interaction is hardcoded to the right hand — no left-handed mode anywhere

- **Impact:** Left-handed players cannot remap anything: the UI/block interaction ray exists only on the right controller (the left has only a teleport arc), so all menu clicking, block breaking (right trigger), block placing (right grip), jump (right A) and block-editing toggle (right B) require the dominant right hand, while the left hand only moves/opens menus. For a Quest store title this is a significant accessibility gap (Meta's VRCs expect handedness consideration), and it makes the game's core verb — pointing and triggering — physically awkward for ~10% of players.
- **Evidence:** Bootstrapper EnsureControllerInteractors (~line 1898): comment 'Builds the native interaction (UI + block targeting, right hand only)' and 'if (role == BlockiverseControllerRole.Right)' gates the interaction ray creation. Prefab confirms: Left Controller's BlockiverseLocomotionRayMediator has interactionRay: {fileID: 0} (line ~170) while the Right Controller's references both rays (line ~6063). BlockiverseInputRig.RefreshCachedActions (lines 307-308) binds break=RightHandMap/Select, place=RightHandMap/Activate. FindControllerHaptics(rig, BlockiverseControllerRole.Right) at bootstrapper line ~2342. BlockiverseXrUiInputConfigurator.Configure (lines 29-35) wires the XRUIInputModule click/scroll only to RightHandMap actions. BlockiverseComfortSettings has no handedness field.
- **Recommended fix:** Add a 'Primary hand' option to BlockiverseComfortSettings (persisted in BlockiverseSettingsPersistence) and build the interaction ray on both controllers, enabling only the dominant one at runtime; swap the break/place/jump/toggle action map lookups and the dominant-hand haptics target based on the setting. ### 5. No stick deadzone on Move/Turn actions; XRI SnapTurnProvider fires on any non-zero vector

- **Severity:** High  |  **Confidence:** Medium  |  **Needs manual verification:** True
- **Files:** `Assets/Blockiverse/Settings/BlockiverseInputActions.inputactions`, `Assets/Blockiverse/Scripts/Editor/BlockiverseProjectBootstrapper.cs`
- **Impact:** Thumbstick drift or sensor noise translates directly into unintended locomotion: slow continuous glide drift (constant low-grade vection, a classic motion-sickness trigger that also keeps the tunneling vignette flickering) and, worse, spurious snap turns — XRI's SnapTurnProvider has no magnitude threshold, so any reading down to Vector2 epsilon (~1e-5) on the right stick picks a cardinal and rotates the camera 45° every 0.5 s debounce window. Quest controllers with worn sticks (a common hardware complaint) would make the game unplayable. The committed action asset and the bootstrapper generator both omit the StickDeadzone processors that Unity's own XRI sample bindings include.
- **Evidence:** Every binding in BlockiverseInputActions.inputactions has "processors": "" (e.g. Move binding id ...024 → <XRController>{LeftHand}/thumbstick, Turn binding id ...052); grep StickDeadzone returns 0 hits in the asset and the bootstrapper (AddControllerMap lines 1051-1052 add Move/Turn with no processors). Verified in packages: SnapTurnProvider.GetTurnAmount (XRI 3.x, SnapTurnProvider.cs lines 193-221) returns m_TurnAmount for ANY input != Vector2.zero via CardinalUtility.GetNearestCardinal — no deadzone; ContinuousMoveProvider reads the raw Vector2. Reading a whole-stick Vector2 binding does not apply the child-axis axisDeadzone processors (Vector2Control.ReadUnprocessedValueFromState reads raw state), and OculusTouchControllerProfile's thumbstick (StickControl via USE_STICK_CONTROL_THUMBSTICKS in ProjectSettings.asset lines 700-701) declares no stick-level deadzone processor.
- **Recommended fix:** Add a 'stickDeadzone' processor (e.g. min=0.15) to the Move, Turn, and UI Scroll bindings in EnsureInputActionSchema/AddControllerMap so the bootstrapper writes it into the asset, and regenerate BlockiverseInputActions.inputactions. Alternatively give SnapTurnProvider input a magnitude threshold by processing in a custom reader.
